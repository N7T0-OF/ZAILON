# Signed in-app updates

ZAILON uses the official Tauri updater. It fetches signed `latest.json`
metadata from the official GitHub release, compares SemVer versions, selects the
matching operating-system and CPU package, validates its signature, and then
hands the installation to the operating system.

## Maintainer setup

The release workflow needs the repository secret named
`TAURI_SIGNING_PRIVATE_KEY`. Its value is the private key generated by:

```powershell
npx tauri signer generate --ci --write-keys "$env:LOCALAPPDATA\ZAILON\keys\zailon-updater.key"
```

Keep that private key backed up in a secure password manager or encrypted vault.
It must never be committed, shared in a chat, or rotated casually: installed
clients use the public key embedded in `src-tauri/tauri.conf.json` to verify all
future releases.

The matching public key is safe to ship in the application. The GitHub Actions
workflow reads the private key only through the encrypted repository secret and
never exposes it in a release asset.

## Channels

- **Stable** uses GitHub's latest non-prerelease release.
- **Beta** asks the public GitHub Releases API for the newest non-draft
  prerelease and then verifies its signed `latest.json` metadata in exactly the
  same way.

Create beta tags using SemVer prerelease syntax, for example `v1.1.0-beta.1`.
The release workflow marks such tags as prereleases automatically.

## Release checklist

1. Change all three application version files.
2. Update `CHANGELOG.md` with user-facing release notes.
3. Commit, push and tag `vX.Y.Z` (or `vX.Y.Z-beta.N`).
4. Confirm the GitHub workflow uploaded installers, `.sig` files,
   `latest.json` and `checksums-sha256.txt`.
5. Install the previous release, select **Settings → Check for updates**, and
   confirm that the new signed release is offered.

The updater creates a local snapshot before downloading, retains three recent
snapshots, and writes the update audit log locally. The application never sends
game paths, mod files or update logs to a ZAILON server.
